Let's make a few changes here. 

The key idea here is we will use a Reflection/self-correction pattern. We will generate insights and graphs from the dataset. Then score these graphs according to the defined rubric, then go and regenerate the graphs again and rescore them, and do this in a loop until the score is Excellent.

The core architecture for this needs to be a "stateful graph", we should use LangGraph, this will allow us to define the loop as a set of nodes and edges where state is preserved. 

The Agent Roles:
- The Orchestrator (The "Brain"):
	Role: Manages the state, decides if the loop should continue, and formats the final output.
	Task: Takes user requirements, coordinates between Visualizer and Scorer, and tracks "Try Count" to prevent infinite loops.
- The Visualizer (The "Coder"):
	Role: Writes and executes Python code (using libraries like Plotly, Seaborn, or Altair).
	Task: Receives the dataset schema and feedback; outputs an image (PNG/SVG) and the code used to generate it.
- The Scorer (The "Critic"):
	Role: A Vision-capable LLM (e.g., GPT-4o or Claude 3.5 Sonnet).
	Task: It doesn't just look at the code; it looks at the actual image. It compares the image against your rubric (e.g., "Is the font readable?", "Are the axes labeled?", "Is the color contrast accessible?") and provides a numeric score and specific critique.

Sometimes the Visualizer might keep making the same mistake, to avoid falling in a loop, the Orchestrator should keep a "Memory" of previous attempts. If the score doesn't improve after 3 iterations, the Orchestrator should change the prompt strategy or stop and ask the user for clarification.
